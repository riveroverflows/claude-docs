# Loopers WIL (What I Learned) 작성 가이드

> 이번 주 루프팩을 통해 **느낀 것, 배운 것, 고민한 것**을 솔직하게 적어보세요.

## WIL은 왜 쓰는 걸까?

매주 배운 걸 기록하거나, 돌아보지 않으면 금방 잊히곤 합니다.

그래서 루프팩에서는 **기술적인 개념 + 내가 느낀 점**을 글로 정리해보는 습관을 만들어 갈꺼예요.

그냥 단순히 **회고**가 아니라, 아래와 같이 학습하고, 체화한 흐름을 나만의 언어로 써보는 게 핵심이에요.

- **무엇을** 중점적으로 학습했는지
- **어떤 것**이 특히 헷갈렸는지
- 결국 어떻게 **이해**하게 됐는지

## 어떻게 쓰면 좋을까?

보통 WIL 은 딱 정해진 형식은 없지만, 이를 작성해 본 분들이 생각보다 많이 없을 거예요.

글을 쓰기 막막하다면, 아래와 같이 써보면 좋아요.

```markdown
# WIL - 1주차 (TDD & 테스트 가능한 구조)

## 이번 주에 새로 배운 것

- 테스트는 “검증 도구”가 아니라, “설계를 이끌어내는 도구”라는 걸 처음 알게 됐어요.
- 테스트 더블(Mock, Stub, Fake 등)의 쓰임과 차이를 실습하면서 체감했습니다.
- 도메인 로직과 의존성을 분리하지 않으면 테스트가 어려워진다는 걸 직접 경험했어요.

## 이런 고민이 있었어요

- 처음엔 그냥 `new UserJpaRepository()`로 객체를 만들었는데, 테스트할 때 DB 연결 때문에 에러가 나서 당황했어요.
- Mock을 썼더니 흐름만 보이긴 하는데, 실제 동작이 불안해서 결국 Fake 클래스를 만들어 사용하게 됐어요.

## 앞으로 실무에 써먹을 수 있을 것 같은 포인트

- 외부 의존성을 인터페이스로 분리하고 생성자 주입하면, 테스트가 정말 쉬워진다.
- 도메인 로직은 가능한 한 순수하게 만들어야 테스트 코드가 깔끔하게 떨어진다.

## 아쉬웠던 점 & 다음 주에 해보고 싶은 것

- 테스트 대상과 협력 객체 사이 책임을 깔끔하게 나누는 데 시간이 오래 걸렸어요.
- 다음 주에는 설계할 때부터 테스트 구조를 먼저 고려해서 더 정돈된 흐름으로 가보고 싶어요.
```

## 이런 흐름이 보이면 좋아요

- **처음엔** 이렇게 생각했는데 → **해보니까** 아니더라 → **그래서** 이렇게 접근해보았다.
- 어떤 개념을 **발제에서 들었는데** → **직접 내 코드에 적용**해보니 이런 느낌이었다.
- **구조가 잘못되었다**고 느낀 이유 → **고친** 이유 → 그 결과 생긴 **변화**

## 톤은 이렇게

| 좋음 👍                              | 안 좋음 👎              |
|------------------------------------|----------------------|
| 처음엔 헷갈렸지만, 구조를 바꾸면서 이해가 됐다.        | 제가 많이 부족해서 죄송합니다.    |
| 이 부분은 아직 잘 모르겠지만 다음에 더 공부해볼 예정이에요. | 다음엔 더 잘하겠습니다. 반성합니다. |
| 이 개념은 이번에 처음 알게 됐는데 생각보다 중요하더라구요.  | ~~이란 무엇인가? ~~는 ~~이다. |

## 팁 몇 가지

- 너무 잘 쓰려고 하지 않아도 돼요. **내가 어떻게 생각했는지**가 더 중요해요.
- 길이 제한은 없어요. 짧더라도 솔직하게!
- 수업 끝나고 **하루 안에** 써보면 기억이 생생해서 더 좋아요.

그냥 솔직하게, 부담 없이 시작해보세요.
