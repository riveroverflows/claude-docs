## 1. 현재 나의 상태

### 배경

- Java 백엔드 개발자
- Kotlin 경험 없음
- Loop:Pak 10주 멘토링 과정 진행 중

### 인지하고 있는 것

- DDD 등 고급 개념에 대한 이해가 부족함
- "왜 이렇게 하는지" 모르고 구조만 따라하면 문제 해결이 어려움
- AI를 잘 활용하려면 기술 학습이 필요함

### 고민했던 것

- "지시만 잘 하면 AI가 다 해주는데 기술을 깊게 파는게 어떤 의미가 있을까?"
- "요즘 추세에 하나의 기술을 깊게 파는 학습을 하는게 맞나?"

## 2. Loop:Pak 교육과정 커리큘럼

### 4단계 구조

| 단계  | 주차    | 주제          | 핵심 학습             |
|-----|-------|-------------|-------------------|
| 1단계 | 1-3주  | AI와 설계의 기본기 | 기초 아키텍처, 기본 설계 원칙 |
| 2단계 | 4-5주  | 데이터 정합성과 성능 | 데이터 관리, 최적화 전략    |
| 3단계 | 6-8주  | 안정성과 확장 구조  | 장애 대응, 시스템 확장성    |
| 4단계 | 9-10주 | 집계와 운영 설계   | 데이터 분석, 운영 관리     |

### 학습 방식

- **주말**: 기술 아티클 리딩, 복습, 핵심 주제 발제, 과제 안내
- **평일**: 라이브 세션, 팀 멘토링, 실시간 Q&A
- **개인**: 구현 과제, 테크니컬 글쓰기, AI 활용
- **팀**: 개념 토론, 설계 공유, 네트워킹

## 3. 교육과정의 핵심 목적

> **"왜 그렇게 판단했는가?"에 답할 수 있는 능력**

이 한 문장이 전부입니다.

- "어떻게 구현하는가?" (How) → AI가 도와줄 수 있음
- "왜 이렇게 판단했는가?" (Why) → 내가 답해야 함

---

## 4. 대화에서 얻은 핵심 인사이트

### DDD 도입을 보류한 이유

팀원의 DDD 코드가 멋져 보였지만, 지금 따라하지 않기로 한 이유:

1. **개념 이해 부족**: "왜 이렇게 하는지" 모르고 구조만 따라하면 문제 발생 시 해결 못 함
2. **학습 우선순위**: TDD, 테스트 작성법, 레이어드 아키텍처가 DDD보다 선행되어야 함
3. **비용-효과**: 완전 분리 구조는 매핑 복잡성이 높아 학습 효율 저하

→ DDD는 `docs/learning/ddd-learning-guide.md`에 정리해두었고, 나중에 학습 예정

### "지시를 잘 하기 위해 학습이 필요하다"

**"지시만 잘 하면 AI가 다 해준다"는 반만 맞는 말.**

- 간단한 CRUD, 보일러플레이트 → AI가 잘 함
- 복잡한 설계, 디버깅, 성능 최적화 → 내가 이해해야 좋은 지시를 할 수 있음
- AI 출력물 검증 → 내가 판단해야 함

**결론**: AI는 "실행"을 도와줌. "판단"은 내 몫.

### 깊이 vs 판단력

```
❌ "기술을 깊게 파야 해" (깊이 자체가 목표)
✅ "이게 왜 이런지 설명할 수 있어야 해" (판단력이 목표)
```

- 깊이는 수단이지 목적이 아님
- 모든 기술을 깊게 파는 게 아니라, **지금 풀어야 할 문제에 필요한 깊이**만큼 파면 됨

### 지금 집중해야 할 것

```
├── TDD (Red → Green → Refactor)
├── Outside-In TDD (Controller → Facade → Domain → Repository)
├── 테스트 레벨 구분 (E2E / Unit / Integration)
├── 레이어드 아키텍처
└── "왜 이렇게 하는가?" 질문하기
```

## 5. 방향을 잃었을 때 체크리스트

### 자가 점검 질문

```
□ 지금 하려는 게 과제 요구사항에 있는 건가?
□ 이걸 왜 이렇게 하는지 설명할 수 있나?
□ AI가 만든 코드를 내가 평가할 수 있나?
□ 테스트가 무엇을 검증하는지 알고 있나?
□ 모르는 개념을 구조만 따라하고 있지 않나?
```

### 방향을 잃었을 때

1. **멈추기** - 무작정 진행하지 않기
2. **PRD 다시 읽기** - `docs/assignments/week*-prd.md`
3. **TDD 순서 확인** - PRD에 Step 1~4가 정의되어 있음
4. **"왜?"를 질문하기** - 이해 안 되면 멘토/헬퍼에게 질문
5. **이 문서 다시 읽기** - 마인드셋 리마인드

### 하지 말아야 할 것

```
❌ 다른 사람과 비교하기
❌ 모르는 개념을 구조만 따라하기
❌ AI가 다 해주니까 대충 하기
❌ "깊이"를 목표로 삼기
```

### 해야 할 것

```
✅ 과제 요구사항에 집중하기
✅ "왜?"를 계속 질문하기
✅ AI 출력물을 평가하는 눈 기르기
✅ 실패를 통해 배우기
```
