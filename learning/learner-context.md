# 학습자 컨텍스트 (Claude 참조용)

> 이 문서는 Claude가 학습자의 상황을 이해하고 적절한 가이드를 제공하기 위한 컨텍스트입니다.

---

## 학습자 프로필

### 배경

- Java 백엔드 개발자
- Kotlin 경험 없음 (이 교육과정이 첫 Kotlin 프로젝트)
- AI 도구(Claude Code) 적극 활용 중

### 현재 상태

- Loop:Pak 10주 멘토링 과정 진행 중
- 1-3주차: AI와 설계의 기본기 학습 단계

### 학습 특성

- DDD 등 고급 개념에 대한 이해 부족 → 개념 없이 구조만 따라하지 않도록 주의
- "왜 이렇게 하는지" 설명할 수 있는 것을 중요하게 생각
- AI를 잘 활용하기 위해 기술 학습이 필요하다는 인식

## Loop:Pak 교육과정 정보

### 커리큘럼 (10주)

| 단계  | 주차    | 주제          | 학습 내용                  |
|-----|-------|-------------|------------------------|
| 1단계 | 1-3주  | AI와 설계의 기본기 | 기초 아키텍처, 기본 설계 원칙, TDD |
| 2단계 | 4-5주  | 데이터 정합성과 성능 | 데이터 관리, 최적화 전략         |
| 3단계 | 6-8주  | 안정성과 확장 구조  | 장애 대응, 시스템 확장성         |
| 4단계 | 9-10주 | 집계와 운영 설계   | 데이터 분석, 운영 관리          |

### 교육과정 핵심 목표

> **"왜 그렇게 판단했는가?"에 답할 수 있는 능력**

- 단순히 "어떻게 구현하는가"가 아닌 "왜 이렇게 판단했는가"를 설명할 수 있는 개발자 양성
- 실무 관점의 판단력 개발

### 개발 방식

- **TDD (Red → Green → Refactor)**: 모든 기능은 TDD 사이클로 개발
- **Outside-In TDD**: Controller → Facade → Domain → Repository 순서로 진행
- **테스트 레벨**: E2E Test, Unit Test, Integration Test 구분

## 학습 방향성과 원칙

### 현재 집중해야 할 것

```
1. TDD 사이클 체화 (Red → Green → Refactor)
2. Outside-In TDD 이해 (Controller → Facade → Domain → Repository)
3. 테스트 레벨 구분 (E2E / Unit / Integration)
4. 레이어드 아키텍처 (interfaces / application / domain / infrastructure)
5. "왜 이렇게 하는가?" 질문하기
```

### 현재 집중하지 않아도 되는 것

- DDD 완전 분리 구조 (Persistence Ignorance)
- CQRS, Event Sourcing
- 복잡한 아키텍처 패턴

### 학습자가 도달한 핵심 인사이트

1. **"지시를 잘 하기 위해 학습이 필요하다"**
    - AI는 "실행"을 도와줌
    - "판단"은 학습자의 몫

2. **깊이보다 판단력**
    - 모든 기술을 깊게 파는 게 아님
    - 지금 풀어야 할 문제에 필요한 깊이만큼만

3. **모르는 개념은 구조만 따라하지 않기**
    - "왜 이렇게 하는지" 모르면 문제 발생 시 해결 못 함
    - 개념 이해 후 적용

## Claude 가이드 지침

### 기본 원칙

1. **과제 범위 내에서 가이드**
    - 현재 주차의 과제 요구사항에 집중
    - 요구사항에 없는 고급 기능/패턴 제안 자제

2. **"왜?"를 설명해주기**
    - 코드를 제시할 때 "왜 이렇게 하는지" 설명 포함
    - 학습자가 판단력을 기를 수 있도록

3. **오버엔지니어링 방지**
    - 현재 필요한 최소한의 구현만 제안
    - 미래의 요구사항을 미리 대비하지 않기

4. **Kotlin-Java 대응 설명**
    - Kotlin 문법 사용 시 Java 대응 패턴 간단히 언급
    - "Java로 치면 이런 거" 형태의 비교 설명

### 하지 말아야 할 것

```
❌ 학습자가 이해하지 못하는 고급 개념 무작정 적용
❌ 요구사항에 없는 기능 추가 제안
❌ "이게 더 좋아요"라며 과한 구조 제안
❌ DDD 완전 분리 등 현재 학습 범위를 벗어난 패턴 적용
```

### 해야 할 것

```
✅ PRD(과제 요구사항) 기준으로 가이드
✅ TDD 순서(Step 1~4) 따르도록 안내
✅ "왜 이렇게 하는지" 설명 포함
✅ 학습자가 모르는 개념이면 개념 설명 먼저
✅ 현재 프로젝트 구조/패턴 따르기
```

### 학습자가 방향을 잃었을 때

1. PRD 다시 확인하도록 안내
2. TDD 순서(Step 1~4) 리마인드
3. 현재 해야 할 단계가 무엇인지 명확히
4. 필요하면 `docs/learning/my-learning-mindset.md` 참조 권유
