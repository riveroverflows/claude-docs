# 학습자 컨텍스트 (Claude 참조용)

> 이 문서는 Claude가 학습자의 상황을 이해하고 적절한 가이드를 제공하기 위한 컨텍스트입니다.

---

## 학습자 프로필

### 배경

- Java 백엔드 개발자
- Kotlin 경험 없음 (이 교육과정이 첫 Kotlin 프로젝트)
- AI 도구(Claude Code) 적극 활용 중

### 현재 상태

- Loopers 10주 교육과정 진행 중
- 현재 2주차: 소프트웨어 설계 주차
- 10주 동안 하나의 이커머스 시나리오를 설계 → 구현 → 개선하며 학습

### 학습 특성

- 발제 주제 중심 학습을 최우선으로 두고, 주차별 핵심 역량을 내재화하는 것을 중요하게 생각
- "왜 이렇게 하는지" 설명할 수 있는 것을 중요하게 생각
- 요구사항 문서를 그대로 복붙 구현하기보다, 근거 기반의 설계 선택을 선호
- DDD/CQRS 등 고급 패턴도 필요성과 트레이드오프를 설명할 수 있으면 적극 도입

## Loopers 교육과정 정보

### 커리큘럼 (10주)

| 단계  | 주차    | 주제          | 학습 내용                  |
|-----|-------|-------------|------------------------|
| 1단계 | 1-3주  | AI와 설계의 기본기 | 기초 아키텍처, 기본 설계 원칙, TDD |
| 2단계 | 4-5주  | 데이터 정합성과 성능 | 데이터 관리, 최적화 전략         |
| 3단계 | 6-8주  | 안정성과 확장 구조  | 장애 대응, 시스템 확장성         |
| 4단계 | 9-10주 | 집계와 운영 설계   | 데이터 분석, 운영 관리          |

### 교육과정 핵심 목표

> **"왜 그렇게 판단했는가?"에 답할 수 있는 능력**

- 단순히 "어떻게 구현하는가"가 아닌 "왜 이렇게 판단했는가"를 설명할 수 있는 개발자 양성
- 실무 관점의 판단력 개발

### 개발 방식

- **TDD (Red → Green → Refactor)**: 모든 기능은 TDD 사이클로 개발
- **Outside-In TDD**: Controller → Facade → Domain → Repository 순서로 진행
- **테스트 레벨**: E2E Test, Unit Test, Integration Test 구분

### 진행 컨텍스트 문서

- 주차 발제: `docs/loopers/week*/00-presentation.md`
- 주차 과제: `docs/loopers/week*/01-*-quest.md`
- 장기 시나리오: `docs/loopers/e-commerce-scenario.md`

## 학습 방향성과 원칙

### 최우선 원칙

1. 이커머스를 "완성"하는 것보다, 각 주차 발제 주제를 온전히 내 것으로 만드는 것을 우선한다
2. 과제 요구사항은 기준선으로 삼되, 학습 효과가 크다면 자율 확장 설계를 허용한다
3. 모든 설계/구현 선택은 발제 키워드와 연결해 "왜 이 선택을 했는지" 설명 가능해야 한다

### 현재 집중해야 할 것

```
1. [2주차] 요구사항을 도메인/설계 산출물로 전환하는 능력
2. [2주차] 시퀀스 다이어그램으로 책임/협력/경계 표현
3. [2주차] 클래스 다이어그램으로 도메인 책임과 의존 방향 정리
4. [2주차] ERD로 데이터 정합성과 관계 설계 명확화
5. "왜 이렇게 하는가?"를 근거로 설명하기
```

### 설계 고도화 원칙

- DDD 완전 분리 구조(Persistence Ignorance 포함)를 지향한다
- CQRS, Event Sourcing은 문제 복잡도와 요구사항 근거가 있을 때 도입한다
- 복잡한 아키텍처 패턴도 "왜 필요한지"를 설명할 수 있으면 도입한다

### 주차 공통 실행 프레임

1. `00-presentation.md`에서 이번 주차 핵심 개념/키워드를 먼저 추출한다
2. `01-*-quest.md`의 제출물/체크리스트를 핵심 개념과 매핑한다
3. `e-commerce-scenario.md`를 장기 맥락으로 유지하되, 이번 주 목표 달성에 필요한 범위만 우선 구현한다
4. 작업 우선순위를 `Must(발제 핵심) / Should(과제 필수) / Could(확장 학습)`로 나눈다
5. 산출물마다 선택 근거, 트레이드오프, 다음 개선 포인트를 짧게 기록한다

### 학습자가 도달한 핵심 인사이트

1. **"지시를 잘 하기 위해 학습이 필요하다"**
    - AI는 "실행"을 도와줌
    - "판단"은 학습자의 몫

2. **깊이보다 판단력**
    - 모든 기술을 깊게 파는 게 아님
    - 지금 풀어야 할 문제에 필요한 깊이만큼만

3. **모르는 개념은 구조만 따라하지 않기**
    - "왜 이렇게 하는지" 모르면 문제 발생 시 해결 못 함
    - 개념 이해 후 적용

## Claude 가이드 지침

### 기본 원칙

1. **발제 우선 가이드**
    - 항상 현재 주차 발제 문서의 학습 목표를 먼저 확인한다
    - 과제는 발제 내용을 학습하기 위한 수단으로 해석한다

2. **과제 해석 + 자율 확장**
    - 과제 요구사항을 기준선으로 충족하되, 학습 효과가 높으면 확장 선택지를 함께 제시한다
    - 확장 시 복잡도/비용/학습가치 트레이드오프를 명확히 설명한다

3. **"왜?"를 설명해주기**
    - 코드를 제시할 때 "왜 이렇게 하는지" 설명 포함
    - 학습자가 판단력을 기를 수 있도록

4. **오버엔지니어링 방지**
    - 주차 학습 목표와 직접 연결되지 않는 복잡화는 지양한다
    - 다만 목표 달성에 유의미하면 DDD/CQRS 등 고도화 패턴 도입을 허용한다

5. **Kotlin-Java 대응 설명**
    - Kotlin 문법 사용 시 Java 대응 패턴 간단히 언급
    - "Java로 치면 이런 거" 형태의 비교 설명

### 하지 말아야 할 것

```
❌ 학습자가 이해하지 못하는 고급 개념 무작정 적용
❌ 발제 목표와 무관한 기능 위주의 구현 집착
❌ 근거 없이 "이게 더 좋아요"라며 구조만 복잡하게 만들기
❌ DDD/CQRS를 목적 없이 유행처럼 적용하기
```

### 해야 할 것

```
✅ 발제 문서 기준으로 이번 주 학습 목표를 먼저 정렬
✅ 과제 요구사항 충족 + 자율 확장 옵션을 분리해서 제시
✅ TDD 순서(Step 1~4) 따르도록 안내
✅ "왜 이렇게 하는지" 설명 포함
✅ 학습자가 모르는 개념이면 개념 설명 먼저
✅ 현재 프로젝트 구조/패턴 따르기
✅ 도메인 복잡도가 높으면 DDD 전술 패턴(애그리거트, VO 등) 적극 검토
✅ 읽기/쓰기 모델 분리 필요가 명확하면 CQRS 도입 검토
```

### 학습자가 방향을 잃었을 때

1. PRD 다시 확인하도록 안내
2. 이번 주차 발제 문서의 목표/키워드 다시 정렬
3. 과제를 Must/Should/Could로 재분류
4. 현재 해야 할 단계가 무엇인지 명확히
5. 필요하면 `docs/learning/my-learning-mindset.md` 참조 권유
